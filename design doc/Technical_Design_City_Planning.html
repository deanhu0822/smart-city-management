Section 2

NVIDIA Stack — 30 Points

2A. The Stack (15 pts)

We use four major NVIDIA tools:

RAPIDS cuDF Core
Load & process all 25 datasets (6M+ rows). GroupBy, merge, feature engineering on GPU.
pandas: ~4 min for LL84 (2.2M rows). cuDF: ~3 seconds. 800x speedup.
RAPIDS cuOpt Core
Optimize DSNY waste collection routes. 20K+ litter baskets, 39 transfer stations, tonnage constraints — Vehicle Routing Problem on GPU.
CPU VRP solvers (OR-Tools): minutes. cuOpt: seconds.
NVIDIA NIMs Core
Local LLM inference (Llama 3.1 8B via NIM container). Site scoring, dispatch recommendations, natural language insights — all on-device.
Cloud LLM = 0 points. NIMs gives OpenAI-compatible API running locally.
NVIDIA Modulus Stretch
Physics-informed neural network for energy demand forecasting. Small PINN respecting thermodynamic constraints.
Traditional forecasting ignores physics; Modulus embeds conservation laws.

2B. The "Spark Story" (15 pts)

"We hold the entire NYC municipal dataset — 6 million rows across 25 datasets — as GPU DataFrames in RAPIDS cuDF, while simultaneously running a 16-billion-parameter language model via NVIDIA NIMs for real-time site analysis, and solving a city-scale vehicle routing problem with cuOpt — all in the 128 GB unified memory of the GB10.

On a typical laptop with 16 GB RAM, you'd have to choose: load the data OR load the model OR run the optimizer. On the Spark, we run all three at once with zero swapping, zero-copy GPU↔CPU transfers, and sub-second response times.

Privacy matters: this is real NYC municipal data about schools, hospitals, and community centers. Running inference locally means no sensitive location data ever leaves the building."
Section 3

Data Pipeline

Stage 1: Ingestion (RAPIDS cuDF)

Python
import cudf # Load 2.2M-row LL84 dataset in ~2 seconds on GPU ll84 = cudf.read_csv("data/raw/ll84_monthly.csv") # vs. pandas: ~45 seconds for the same file

All 25 datasets loaded into GPU memory as cuDF DataFrames. Total footprint: ~8 GB.

Stage 2: Feature Engineering (cuDF + cuSpatial)

Python
import cuspatial # Find EV stations within 500m of each building nearby_ev = cuspatial.pairwise_point_distance( site_points, ev_points ) sites["ev_within_500m"] = (nearby_ev < 500).sum(axis=1)

Key operations: BBL join, district aggregation, spatial proximity, time-series features, waste composition overlay.

Output: unified_sites.parquet (4,268 × 45 features) + district_waste.parquet (59 × 30 features)

Stage 3: LLM Scoring (NVIDIA NIMs)

Python
from openai import OpenAI client = OpenAI( base_url="http://localhost:8000/v1", # NIM local api_key="not-needed" ) response = client.chat.completions.create( model="meta/llama-3.1-8b-instruct", messages=[ {"role": "system", "content": SCORING_PROMPT}, {"role": "user", "content": batch_json} ], temperature=0.1 )

Batch size: 20 sites/call. ~214 batches. ~2 sec/batch → ~7 min for all 4,268 sites.

Stage 4: Route Optimization (cuOpt)

Python
from cuopt import routing problem = routing.DataModel() problem.set_locations(stations + districts) problem.set_demand(district_tonnage) problem.set_vehicle_capacities(station_caps) solver = routing.SolverSettings() solver.set_time_limit(10) # seconds solution = routing.Solve(problem, solver)

Stage 5: BESS Dispatch Simulation

24h charge/discharge simulation: ConEd TOU rates, solar curves, EV patterns. 365 days × 50 sites = 18,250 simulations. On GPU: ~30 seconds total.

Section 8

Risk Mitigation

NIM container setup fails Medium
Fallback: llama.cpp with CUDA backend. Still local, still GPU-accelerated.
cuOpt installation issues Medium
Fallback: cuDF distance matrix + greedy assignment. Still GPU-accelerated.
LL84 dataset too large to download Low
Pre-download to USB drive before kickoff.
Dashboard crashes during demo Medium
Pre-recorded video backup. Pre-run pipeline so dashboard only reads cached files.
Modulus setup too complex High
Stretch goal. Skip by Sat 6 PM if not working. cuDF + cuOpt + NIM = enough for full NVIDIA points.
